# V1 Scope Lock

## Purpose

This document freezes the current product scope for AlgebraTrust v1.

Its job is to prevent scope drift while preserving a clear path for later
expansion.

This document is about **product scope**, not low-level implementation details.

Use it to decide:

- what the current product is
- what the current product is not
- what the first instructional lane includes
- what later expansion is allowed to look like

When there is tension between a broader idea and the current scope, this
document wins unless explicitly revised.

---

## Scope Model

AlgebraTrust has three different scopes.

### 1. Vision scope

A semester-long college algebra workspace.

### 2. Architecture scope

A system designed so additional instructional lanes can be added later.

### 3. Current implementation scope

A narrow first lane that proves the product works.

These are not the same thing.

Do not confuse semester-long vision with semester-wide implementation.

---

## Current Product Definition

AlgebraTrust v1 is a **visual, trust-checked college algebra workspace** with a
narrow first implementation focus.

### Current implementation focus

The current deep support lane is:

- quadratic equations
- quadratic functions
- factoring support where needed
- graph-based understanding of parabolas

This is the first lane because it offers:

- strong visual teaching value
- strong student usefulness
- deterministic validation opportunities
- reusable foundations for later units
- a clear demo path

---

## Current North Star

The broad long-term direction is:

> a semester-long college algebra workspace that helps students ask questions
> privately, understand concepts visually, and work through problems without
> needing paper

The current actionable north star is:

> build a visually strong, trust-checked quadratic learning workspace that
> proves the larger product direction

## V1 Deep Support

The following topics are in deep support for v1.

### Quadratic equations

Includes:

- identifying quadratic equations in common forms
- solving common quadratic cases
- factoring-first solution flow
- interpreting roots as solutions
- connecting factor form to solutions
- checking common factoring attempts

### Quadratic functions

Includes:

- graphing parabolas
- identifying vertex
- identifying roots / x-intercepts
- identifying axis of symmetry
- identifying opening direction
- relating equation form to graph shape

### Factoring support

Includes:

- factoring support only when it directly helps the quadratic lane
- factoring trinomials in common cases
- linking factorization to roots

### Graph-based understanding

Includes:

- seeing equation and graph together
- visual explanation of roots, vertex, and axis
- basic coefficient influence on the parabola

## V1 Core Interactions

These interaction types are part of the locked v1 product shape.

### Solve with me

This is the **hero flow**.

The student enters a quadratic or asks for help solving one.
The system guides them through a structured solving flow and connects the steps
to the graph.

### Visualize this

This is a **supporting interaction**.

The student asks to see a quadratic or understand a graph-related idea.
The graph is used as a primary explanation surface.

### Check my work

This is a **supporting interaction**.

The student submits an answer or a narrow supported intermediate step.
The system validates it and explains the result.

### Explanation behavior

Explanation is not a separate free-floating mode.
Explanation is woven into solve, visualize, and check.

## V1 UI Shape

The UI shape is locked for the first build.

### Layout

- two-panel workspace

### Left panel

- question input
- current explanation
- next step prompt
- action buttons
- math input / rendered preview

### Right panel

- graph area
- trust indicators
- key math facts

### UX rule

The product should feel like a structured guided workspace.

It should **not** feel like:

- a generic chatbot
- an unrestricted conversation transcript
- a developer dashboard
- a feature playground

---

## V1 Trust Model

The trust model is part of the current product definition.

### Locked principle

The system should not merely answer.
It should show which checks passed.

### Current truth model

The deterministic math layer is the source of mathematical truth.

### Language layer role

The language layer may help with:

- question interpretation
- tutoring tone
- explanation style
- guided flow
- light personalization

The language layer is **not** the source of mathematical truth.

## V1 Graph Scope

The graph scope must remain narrow in the first version.

### Included graph features

- vertex
- roots
- axis of symmetry
- opening direction
- coefficient influence
- a, b, c slider behavior when the graph milestone is reached

### Excluded graph features for now

- 3D visuals
- broad multi-topic graphing
- advanced graph comparison systems
- highly custom exploratory sandboxes unrelated to the current lane

---

## V1 Math Scope Limits

The current math scope is intentionally limited.

### Supported now

- common quadratic input forms
- factorable quadratics
- direct graph interpretation for quadratics
- narrow supported check-my-work cases
- trusted extraction of key graph facts

### Not promised yet

- broad algebra coverage
- arbitrary symbolic manipulation
- deep equivalence checking across many student step styles
- full natural-math parsing
- full-semester deterministic support

---

## Explicit Out of Scope for V1

The following are out of scope unless explicitly promoted later.

### Product breadth

- full-semester deep support
- broad coverage of all chapters
- systems as a current deep lane
- exponentials/logarithms as a current deep lane
- matrices
- conics beyond what is useful for current quadratic understanding
- sequences, induction, and probability

### Infrastructure

- auth/login
- database-backed history
- analytics dashboards
- multi-provider BYOK infrastructure
- complex account settings
- large-scale ingestion systems

### Interaction scope

- open-ended chatbot behavior
- broad unrestricted “ask anything in math” behavior
- multi-agent UI theatrics
- handwriting recognition
- voice-first flows
- 3D interaction systems

### Technical extras

- speculative abstractions for future lanes
- premature generic engines
- infrastructure that does not support the current lane
- major graphing investment before the graph spike milestone

## Semester Vision Without Scope Drift

The semester-long vision remains valid.

That does **not** mean all textbook units should be deeply implemented now.

### Allowed interpretation

The repo may describe AlgebraTrust as a semester-long college algebra workspace
**as long as** it is also clear that the current deep implementation focus is
the quadratic lane.

### Not allowed interpretation

The repo should not imply that all major course units are already deeply
supported when they are not.

## Expansion Ladder

Expansion is allowed only in deliberate stages.

### Stage 1 — current

Deep support for:

- quadratic equations
- quadratic functions
- factoring support where needed
- graph interpretation for parabolas

### Stage 2 — next likely expansion

Possible additions after Stage 1 is strong:

- linear equations
- graph basics
- function foundations
- transformations

### Stage 3 — later lanes

Possible later additions:

- systems
- inequalities
- exponential and logarithmic topics
- additional course units as justified

This is an expansion ladder, not a commitment to implement every later stage
immediately.

---

## Change Acceptance Rule

A new idea may be accepted into the current scope only if it does at least one
of the following:

- strengthens the current quadratic lane
- improves the shared workspace foundation
- improves trust, clarity, or usability in the current product shape
- clearly prepares the next instructional lane without destabilizing the
  current one

Delay a new idea if it:

- widens the curriculum early
- adds infrastructure without clear current value
- increases maintenance burden for a future that is not yet implemented
- turns the app into a generic chatbot
- weakens the clarity of the current demo path

---

## Decision Questions

Before adding a feature, ask:

### 1. Does it strengthen the current deep lane?

If no, delay it.

### 2. Does it support the current hero flow?

If no, delay it unless it materially strengthens the shared foundation.

### 3. Does it add significant new infrastructure?

If yes, require stronger justification.

### 4. Does it make the product feel broader but weaker?

If yes, reject it.

### 5. Does it preserve the structured workspace identity?

If no, reject it unless the product direction is being deliberately revised.

---

## Practical Rule for Contributors

Contributors should assume the following until this document changes:

- semester-long vision
- quadratic-lane implementation
- structured workspace, not chatbot
- deterministic truth model
- trust-checked output
- depth before breadth

Do not widen scope because a feature seems "small."
Small scope expansions often create large downstream maintenance obligations.

---

## Summary

AlgebraTrust is allowed to have a broad educational vision.

AlgebraTrust v1 is not allowed to become broad implementation-by-accident.

The current locked reality is:

- semester-long vision
- narrow first deep lane
- quadratics first
- structured workspace
- trust-checked output
- expansion only after the first lane is strong

When in doubt:

**finish the current lane cleanly before expanding the product**
